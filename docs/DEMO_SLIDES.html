02Why SDD, why Spendly

  • Objective: internalize a discipline — never write feature code before a written, reviewable spec with acceptance scenarios.
  • Spendly = deliberately boring CRUD. Everyone understands "add / edit / delete / total an expense."
  • A familiar domain keeps attention on the process, not the problem.
Chosen for

An unremarkable domain — so SDD stays the visible thing.

03The SDD workflow, as built here

proposal.md

The why & what changes — for a reviewer.

design.md

Decisions & trade-offs — for an implementer.

spec.md

Requirements as WHEN / THEN scenarios.

tasks.md

Ordered, checkable to-do list.

openspec/specs/= current truth · changes/archive/= how we got here

Spec written & validated first → tasks → code → openspec validate + pytest → archive.

04AI Contribution Summary

6 features (capabilities) implemented. Claude gave a solid foundation; I did the engineering to make the specs production-ready. (estimated split)

Generated by Claude · ~60%

  • Core functional requirements across all 6 capabilities
  • Happy-path CRUD, auth & dashboard flows
  • Baseline WHEN/THEN scenarios & the data model
  • Initial route, template & test scaffolding

Added / Refined by Me · ~40%

  • Missing scenarios & edge cases Claude omitted
  • Security hardening & production validations
  • Ownership / no-disclosure requirements
  • Tightened every acceptance criterion to be testable

Case study · Authentication

Claude generated (foundation): registration, login, session handling, protected routes (login_required), basic input validation.

I added / refined (production-ready): generic login error (blocks account enumeration), password hashing + min-length rule, duplicate-email & blank-field handling, session-fixation guard (session.clear()).

Identified for production (roadmap): RBAC · refresh-token flow · password reset · email verification.

07The working software

Auth

Login as seeded demo user. Password is a Werkzeug hash — DB never sees plaintext.

Total, count & per-category breakdown, computed server-side, scoped to your rows.

Create + Edit

Same server-side validation function — one code path, two routes.

Delete

POST-only, owner-scoped. Another user's row → 404, no disclosure.

08Architecture — thin by design
